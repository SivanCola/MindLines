# AGENTS.md

本文件适用于其所在目录及所有子目录。处理 Mindline 项目文件时，必须遵循以下规则。

## 沟通

- 默认使用中文回复，除非用户明确要求使用其他语言，或代码、日志、错误信息本身更适合保留英文。
- 回复要直接、具体、可执行；避免空泛说明。
- 解释方案时，尽量说明产品价值、交互影响、技术取舍和验证方式。
- 当用户用截图红框指出问题时，不要只按红框局部修补；先判断该区域属于哪一类产品语言，再搜索同类组件，尽量统一处理。

## 开发方式

- 开始实现前，先阅读相关代码、配置和已有约定，再决定改法。
- 优先沿用项目现有架构、命名、样式、组件和工具链；不为了“看起来更高级”引入不必要的新抽象或依赖。
- 改动范围要收敛到当前任务，避免顺手重构无关代码。
- 遇到已有未提交改动时，默认认为是用户的工作；不要回滚、覆盖或格式化无关文件。
- 代码风格以可读为先：命名自解释优先，注释只写必要上下文；三个清晰相似行好过一个过早抽象。
- 对涉及行为变化、数据结构、状态管理、持久化、权限、网络请求或用户输入的改动，要考虑测试、边界条件和失败场景。
- 处理异步生成、流式输出、取消和错误状态时，要以明确事件收尾清理 UI 状态；收到 `done`、`aborted`、`error` 后，都应确保对应生成态能被释放，避免输入框永久禁用。
- 对本机偏好设置优先使用已有设置 hook 和 `localStorage` 模式；除非用户明确要求同步到工作区，否则不要把界面偏好、快捷键偏好写入工作区数据。
- 可以参考 GitHub 上优秀开源项目的设计和实现思路，但只借鉴模式和信息组织；注意许可证、依赖成本和与本项目风格的一致性，不直接复制不明来源代码。
- 当任务明显匹配当前环境中的 skill 时，应先阅读对应 `SKILL.md`，再按其中流程执行。

## 数据与存储

- 区分项目知识资产、本机设置和内部元数据：阶段、话题、消息、总结等应保存在项目入口目录可见的 `topics/`、`phases/` 等目录；manifest、索引等内部元数据可放在项目 `.mindline/`；最近打开记录、模型供应商配置、API Key、界面/快捷键偏好等本机信息应保存在 `~/.mindline/` 或本机偏好存储中。
- 用户级模型配置目录应按领域聚合且保留密钥隔离：模型供应商配置放 `~/.mindline/model-providers/config.json`，密钥放 `~/.mindline/model-providers/secrets/`；不要在 Mindline 新结构中继续使用旧品牌或旧项目名目录，例如 `.groupai/`。旧目录只作为兼容迁移来源。
- 做存储路径调整时必须提供向后兼容迁移：旧目录只作为迁移来源，迁移成功前不要删除；真实测试数据迁移应保留旧目录作为备份，并在最终说明新旧路径。
- 从 `{project}/.mindline/topics` 或 `{project}/.mindline/phases` 迁移到项目根目录可见 `topics/`、`phases/` 后，应清理隐藏目录里的旧副本，避免用户看到根目录数据和隐藏数据不一致。
- `manifest.phasePaths` 和 `manifest.topicPaths` 是可读目录结构的索引，任何移动或重命名阶段目录、话题目录的操作，都必须同步更新两个索引和磁盘目录。
- 重命名阶段 slug 时，不只更新该阶段的 `phasePaths`；还要把该阶段下所有话题的 `topicPaths` 前缀一起改到新阶段路径，否则后续 `addMessage`、生成总结、导出等写路径会落到旧目录。
- 读路径可以保留候选目录兜底用于迁移，但写路径必须只依赖最新 manifest；修复存储路径 bug 时要补“迁移/重命名后继续写入”的回归测试。
- 默认阶段是特殊系统容器，路径固定弱化为 `default-phase`；除非先改数据模型，否则不要把它纳入普通 `phaseIds` 排序或普通阶段 slug 去重逻辑。

## 产品设计原则

- 尽可能从产品经理和交互设计师的角度思考页面与功能：明确目标用户、核心场景、主路径、异常路径和反馈机制。
- Mindline 是频繁使用的工具型桌面应用，应优先保证效率、稳定、密度和可扫描性，而不是堆叠装饰性视觉元素。
- 视觉层级优先于单纯“好看”。标题、当前对象、主操作应是第一层级；数量、路径、状态说明、辅助文案应保持可扫描但低权重，不能和主内容抢注意力。
- 区分“状态”和“动作”：话题数、已保存、当前启用、可用等是状态；移动、复制、删除、导出等是动作。状态不应做得像按钮，动作也不应和状态混在同一视觉层级里。
- 空数量或无数据状态要减少噪音。子项消息数为 0 时优先隐藏或保留空位；分组没有归入任何内容时使用明确空状态文案，不要展示一列显眼的 `0`。
- UI 调整后要同时回看浅色和深色主题。菜单、计数、badge、按钮 hover/active 这类小元素在不同主题下容易出现对比过强、过弱或面积过重的问题。
- 同类元素必须共享产品语言：同一类入口、按钮、菜单、快捷键 keycap、空状态、计数、toolbar 不应在不同页面呈现多套视觉。

## App 框架与侧栏

- 左侧应明确拆分为 activity bar 和 context sidebar：activity bar 只表达一级模块切换，使用图标按钮、固定命中盒和明确选中条；context sidebar 承载当前模块内容，例如工作区卡片、阶段目录、最近话题、回收站操作和设置分类。
- 顶栏、侧栏和正文头部要有明确分工，避免重复表达同一信息：侧栏工作区卡片是工作区入口；顶栏 breadcrumb 只负责层级导航和返回上级；正文头部才是当前对象的主标题、编辑入口和元信息区域。
- 工作区选择器应作为侧栏顶部的稳定上下文，不要在每个模块说明卡里重复放大工作区名；工作区目录名这类上下文应放在工作区卡片或低权重元信息中。
- context sidebar 顶部工具条要克制：只放当前模块名、图标、数量或主要局部动作；不要用大卡片重复正文头部已经表达的标题和说明。
- 概览、阶段、回收站、设置、备份等一级模块进入正文后，正文头部属于同一类产品语言，应统一内容结构、高度、padding、分隔线、副标题样式和动作区规格。
- 如果某个入口没有主动作，也要保持头部高度和对齐稳定，不要因为按钮数量不同导致页面起点漂移。
- 侧栏导航项是频繁点击区域，应禁用无意义文本选择，避免连续点击一级入口或折叠控件时出现蓝色文本选区。若需要文本可复制，应只在具体内容区保留选择能力。

## 阶段、话题与概览

- 侧栏导航要明确区分一级入口和二级对象。概览、阶段、回收站、设置这类一级入口代表视图或导航分组；具体阶段、话题是二级对象。
- 一级“阶段”不应直接打开某个具体阶段，避免一级入口和二级对象同时像“当前项”；它更适合进入阶段概览，并作为二级阶段列表的折叠/展开控制。
- 侧栏分组折叠行为要符合层级预期：点击一级“阶段”进入阶段概览时，应能收起或展开整个二级阶段列表；展开二级阶段列表时，阶段下的话题默认保持收起。进入某个具体阶段或话题时，再自动展开对应上下文。
- 阶段（容器）和话题（内容）必须使用两套不同的视觉模型；不要只靠缩进区分层级。
- 树状层级优先用“整组导轨 + 子项分叉线”表达父子关系，不要只画零散短线或只调背景色。
- 子话题消息数属于辅助信息：默认使用更小字号、更轻字重、低对比度和稳定右对齐；不要默认加明显胶囊背景。阶段总数这类分组摘要可以使用浅色 badge，但子话题消息数不应比话题名更突出。
- 概览页左侧 context sidebar 和正文不应同时展示“最近话题”。如果最近话题用于快速导航，应放在左侧；正文保留统计和阶段目录等概览内容。
- 概览页阶段目录不应使用大面积卡片堆叠。优先使用紧凑列表或表格化行：左侧是阶段标识和标题，中间是日期/状态，右侧是低权重话题数或操作，减少纵向浪费。
- 阶段目录的选中态应轻量，不能比正文头部和主操作更抢眼；行高、分隔线和 hover 反馈要稳定。
- 话题页和阶段页的正文头部应保持同一产品语言：去掉“话题”“阶段”这类低信息量小标签，主标题直接显示当前对象名称；副信息展示真正补充上下文的元数据。话题更适合显示“消息数 · 创建于”；阶段更适合显示“话题数 · 开始于/已结束”。
- 新建阶段或话题优先使用“创建即命名”：在目标树位置插入临时草稿节点，自动聚焦输入框，确认后才调用创建 API，不要先落盘 `New` 再让用户二次重命名。
- 来自不同入口的新建动作（侧栏菜单、详情页按钮、概览页按钮、快捷键）要汇聚到同一套草稿请求逻辑，避免某些入口继续生成占位名称。
- `New` 只能作为底层服务的兜底容错，不应作为正常 UI 创建路径的用户可见默认名称。

## 回收站、备份与设置

- 回收站左侧 context sidebar 不要重复展示一个“回收站”选中项；activity bar 已经表达当前模块，context sidebar 只需要模块工具条、数量和危险操作入口。
- 回收站正文负责展示可恢复对象列表；左侧只放低频操作，例如清空回收站。危险操作默认应低权重展示，并保留必要确认。
- 备份页左侧 context sidebar 应服务于状态和入口，不要用大说明卡抢占空间；备份仓库路径等长信息默认单行截断，需要完整内容时用 tooltip/popover。
- 设置页应采用“分类导航 + 分组设置项 + 明确操作”的信息架构；参考 VS Code、GitHub Desktop、Joplin、GitHub Primer、Carbon 等成熟工具/设计系统时，只借鉴交互模式和信息组织，不直接复制代码。
- 区分“开关”和“分段选择”：开关只用于真正的启用/关闭状态；语言、接入类型、协议等互斥选项应使用 segmented control 或合适选择器，语义不能伪装成 on/off。
- 管理列表中应先展示当前状态和已有配置，再提供新增入口。低频新增表单默认可折叠；危险操作弱化展示或放入更多菜单。
- 长路径、长说明和配置值默认单行截断；需要查看完整内容时，优先使用 hover/focus 浮层或 tooltip，不要原地展开导致设置行高度变化。
- 同一个视觉信息单元只能保留一套 tooltip 来源和一套内容格式。不要让父容器和子元素同时设置不同 `title`，也不要让原生 `title` 与自定义 tooltip/popover 同时负责同一段数据。

## 按钮、菜单、快捷键与资料篮

- 同一页面内含义相近的内联操作按钮应复用同一套基础规格：高度、最小宽度、圆角、图标尺寸、图标与文字间距、字号、字重和 hover/focus 反馈要一致。语义差异只通过 primary、danger、ghost 等变体表达。
- 复制类操作应采用固定尺寸按钮或 icon button；点击后优先通过图标变化、tooltip、aria-label 或轻量 toast 反馈成功，不要用更长文案撑大按钮。
- 资料篮面板标题和右侧工具按钮要在同一条基线上；同一组 icon button 必须固定宽高和 `min-width`，避免因为图标、边框或 collapse 按钮不同造成高度不齐。
- 顶部搜索框里的快捷键提示应与设置页快捷键列表使用同一套 keycap 风格：统一边框、背景、圆角、字重、阴影和组合键分隔符。不要让全局搜索入口和设置说明出现两套快捷键视觉语言。
- 菜单、二级菜单、popover、toolbar 这类嵌套交互区域，禁止用 `.某容器 button`、`.header-actions button`、`button:not(...)` 之类宽泛选择器表达业务语义或主按钮样式；应使用直接子选择器或专用语义 class。
- 菜单项状态要区分“默认、hover/focus、当前选中、危险操作、不可用”。未选菜单项默认保持透明或极弱背景；当前项优先用 check/icon + 轻量背景表达，不要把所有候选项做成大面积填充色卡。
- 全局按钮 hover/focus 只能作为低优先级兜底。不要在全局 `button:hover` 后连续堆多个 `:not(...)` 来排除状态；需要排除状态时优先使用 `:where(:not(...))` 保持低优先级，让组件自己的 class 自然胜出。
- 填充色主按钮的 normal、hover、focus-visible 状态应同时明确 `border-color`、`background`/`background-color` 和 `color`，避免通用 hover、主题变量或背景简写/长写混用造成不可读状态。
- 同一移动/选择目标入口要共享菜单语言。侧栏右键菜单、详情页更多菜单、二级阶段选择器等相同任务，应统一宽度、行高、圆角、阴影、搜索框、当前态和 hover 规则。

## 输入、快捷键与生成

- 同一类输入体验必须跨入口保持一致。主聊天框、上下文篮讨论框等“输入后提交给模型”的 textarea，应共享同一套提交/换行规则和设置项。
- 聊天类输入的推荐默认行为是：`Enter` 发送，`Shift + Enter` 换行，`Cmd/Ctrl + Enter` 作为显式发送备用；如果提供“按 Enter 发送”开关，关闭后 `Enter` 应只换行，`Cmd/Ctrl + Enter` 仍可发送。
- 处理键盘提交时要兼顾输入法组合状态、修饰键和事件冒泡：`isComposing` 时不得发送；局部输入框处理 `Enter`/`Cmd/Ctrl + Enter` 后应阻止默认行为并阻止冒泡，避免误触发全局快捷键或其他输入框提交。
- 快捷键设置页不仅要提供开关，还要展示当前实际生效的快捷键说明；当开关改变发送/换行行为时，说明文案应同步变化。
- 浮层从输入栏或底部工具栏向上展开时，必须同时考虑内部滚动和外部可视区域裁切。不要只设置 `scrollTop = 0`；还要控制 `max-height`、锚点方向、可视区域余量和首次绘制时机。
- listbox 的“当前选中项”和“正在新增项”是两个不同状态。点击“添加角色/添加画像”这类新增入口时，旧选中项应退出 active，新增入口进入临时 active；保存后才切换为新画像的正式选中态。
- 画像、角色、预设等会影响 prompt 的控件必须把显示值、默认策略、用户补充内容和实际模型请求绑定到同一状态来源。

## Markdown 与总结

- Markdown 渲染必须作为全局能力维护。渲染端应统一经过 `MarkdownContent`，不要在不同页面各自实现 Markdown 渲染。
- 修改 Markdown CSS 时要理解 `react-markdown` + `remark-gfm` 的实际 DOM 结构。GFM task list 会产生 `li.task-list-item > input + 多个文本/inline/block 节点`；如果直接把 `li` 设为 grid/flex 而不包裹 checkbox 后内容，会把文本拆成多个布局项。
- task list 应把 checkbox 后的所有内容包成稳定内容容器，并给普通 `li` 与内容容器设置 `min-width: 0` 和合理换行规则。
- 总结区的滚动行为必须符合用户预期：除非正在生成总结，需要把生成进度滚入视野；普通进入已有总结的页面时应从页面顶部开始展示，不要因为已有 summary 自动 `scrollIntoView` 跳过正文头部。
- 话题已有总结时，结论默认展开，避免用户进入后只看到信息量很低的折叠摘要。

## Tooltip 与可访问性

- 项目内 tooltip 统一走 `GlobalTooltip` 和 `data-tooltip`，不要为局部按钮另写一套 tooltip 样式或定位逻辑。
- 左侧 activity bar 的 tooltip 固定向右弹出，并做 viewport clamp；顶部和内容区按钮可继续使用全局上/下自动翻转。
- tooltip 样式应使用主题 token（如 `--tooltip-bg`、`--tooltip-fg`、`--tooltip-border`、`--tooltip-shadow`），避免固定白底。
- 图标按钮必须有稳定命中盒，例如 activity bar 入口使用固定尺寸；不要只让 SVG 图形区域触发 tooltip。
- 带 `data-tooltip` 的按钮内 SVG 应避免接管 pointer 事件，让 hover/focus 命中统一落在按钮本体上。
- `GlobalTooltip` 的定位和显隐状态更新必须幂等：如果 next position 与 current 完全一致，返回 current，不要创建新对象，否则在 React effect/layout effect 中容易触发 `Maximum update depth exceeded`。
- 自定义控件要保留可访问语义，例如 `aria-label`、`aria-expanded`、`aria-selected`、`role` 和键盘 focus 行为；不要只用颜色表达当前状态。

## 工具链与验证

- 本项目当前工具链要求 Node 22；跑测试或构建前先确认 `node -v` 和 `which node`，避免误用系统旧版本 Node。
- Codex shell 可能读取 `~/.zprofile` 而不是只读取 `~/.zshrc`；如果本机用 nvm，确保 `~/.zprofile` 也初始化 nvm 并 `nvm use default`。
- 遇到 Vitest/Vite 启动阶段报 `node:util` 缺少 `styleText` 这类错误，优先检查 Node 版本，不要先假设是业务测试失败。
- 完成代码改动后，尽量运行与改动范围匹配的检查，例如类型检查、lint、单元测试、构建或关键页面手动验证。
- 每次完成项目文件修改后，默认执行项目编译/构建检查，并启动或重新打开项目供用户查看；如果当前环境无法执行或打开，必须在最终回复中说明原因。
- 启动 Electron 开发版时，要确认 Vite 端口和 `VITE_DEV_SERVER_URL` 一致。项目默认脚本可能监听 `5173`，如果该端口已被占用而 Vite 自动切到其他端口，Electron 仍可能连接旧服务；此时应使用显式端口和 `VITE_DEV_SERVER_URL` 启动，或先处理占用端口。
- 运行 `npm run build` 会重建 `dist`，可能导致正在运行的 Electron 开发窗口退出；构建后如需用户查看，应重新启动或确认开发版仍连接到最新代码。
- `@browser-use` 打开的 `http://127.0.0.1:*` 是普通浏览器环境，不等同于 Electron 桌面壳；它适合验证 Web 入口和响应式提示页，但不能验证依赖 `window.groupAI`、系统 Dock、原生窗口、菜单等 Electron 能力。
- 开发模式下 macOS Dock 图标不会自动使用 `electron-builder` 的 `build.icon` 配置；需要在主进程显式设置 `BrowserWindow.icon` 和 `app.dock.setIcon(...)`。

## 回归清单

- 修改一级入口头部样式后，要同时进入概览、阶段、回收站、备份、设置回看正文头部区域，确认标题、副标题、右侧动作区、底部分隔线、首个内容块起始位置保持一致。
- 修改侧栏一级/二级导航后，要回归：打开应用默认页、点击概览、点击阶段进入阶段概览、反复点击阶段折叠/展开二级阶段列表、点击具体阶段、点击阶段下话题、点击回收站和设置。
- 修改概览页后，要确认左侧最近话题和正文内容没有重复，阶段目录密度合理，空阶段、默认阶段、长标题和话题数在浅色/深色主题下都可读。
- 修改资料篮后，要确认标题、左侧操作按钮、右侧折叠按钮和面板内容在同一视觉节奏里；按钮高度、命中盒、禁用态和 tooltip 不应漂移。
- 修改快捷键样式后，要同时回看顶部搜索入口和设置页快捷键列表，确认 keycap 规格、组合符号、浅色/深色主题对比度一致。
- 修改 Markdown 渲染或样式后，要至少回归普通列表、编号列表、GFM task list、嵌套列表、长中文文本、长英文/URL、表格和代码块。
- 修改总结页滚动、展开/折叠或生成状态后，要分别验证：首次进入已有总结、正在生成总结、总结刚完成、切换到另一个话题/阶段、从侧栏重新进入当前页面。
- 修改画像、角色、预设、自定义菜单或其他向上展开的 popover 后，要回归：首次打开是否从第一个可选项开始、当前项在列表中间/底部时是否错误滚到当前项、长列表是否被窗口顶部裁切、打开新增入口时 active 是否转移、切换不同画像时设定内容是否随之变化、保存后是否持久化。
- 验证过程中如果 `typecheck`、测试或构建报出看似无关的错误，不要直接忽略。应先定位是否为本次改动链路遗留、并发 watch/构建读取到中间状态，或确实是已有脏工作区问题；最终不能把项目留在不可类型检查或不可构建的状态，除非明确说明这是用户已有改动造成且本次任务无法安全处理。

## Git 与 AI 协作安全

- 每次让 Cursor、Codex 或其他 AI 工具进行大改前，必须先运行 `git status`，确认当前分支、远端跟踪状态、已修改文件和未跟踪文件。
- 不要在存在未提交改动时切换分支，尤其不要从 `main` 切到 `dist`、`release` 等发布/产物分支；确实需要切分支时，先提交、stash 或明确备份当前改动。
- 每完成一个可运行、可验证的阶段就提交一次 Git，不要连续多天堆积大量未提交代码；提交前至少跑 `npm run typecheck` 和相关测试。
- 至少每天推送一次 GitHub；本地提交不是完整备份，机器故障、IDE 操作或工具误操作仍可能造成丢失。
- 大改动必须使用短生命周期分支，例如 `codex/sidebar-refactor`；确认无误后再合回 `main`。
- 重要但被 `.gitignore` 忽略的文件不能只存在于工作区；要么纳入 Git，要么写入明确的外部备份位置。
- 使用 Cursor 前关闭或谨慎使用“自动丢弃”“自动切换”“自动同步”等高风险能力；任何 `reset`、`checkout`、`clean`、批量删除或覆盖操作都必须先确认影响范围。
- 一旦发现代码消失或文件异常，不要继续编辑或切分支；先检查 `git reflog`、`git stash list`、IDE Local History、Cursor/Codex 会话记录和临时备份，再决定恢复方案。
- 提交前不要只看测试结果，还要复核 `git diff --cached --stat` 和 `git diff --cached --check`；涉及密钥、远端、附件或备份逻辑时，额外扫描 `token`、`secret`、`password`、`api_key` 等敏感字样。
